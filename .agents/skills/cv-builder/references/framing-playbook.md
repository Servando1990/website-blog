# Framing Playbook

Use this file after the user has provided the job description or confirmed the overall bucket.

## First Question

Ask this first when the framing is not already obvious:

`Should I frame this for a technical role or a business/client-facing role?`

Use the two buckets below:

- `technical`: Software Engineer, ML Engineer, AI Engineer, platform-heavy builder roles
- `business`: Solutions Engineer, AI Consultant, founder/operator, customer-facing delivery roles

## Technical Framing

### Narrative

Position Servando as a senior hands-on builder who has designed and shipped ML, data, and AI systems across multiple industries. Emphasize implementation depth, architecture, APIs, orchestration, data contracts, ranking logic, optimization, analytics, and production delivery.

### What To Emphasize

- Python, FastAPI, Pydantic, orchestration, APIs, workflow design
- LLM systems, structured extraction, scoring pipelines, reasoning layers
- React or TypeScript only when it strengthens full-stack credibility
- infrastructure or deployment patterns when they are clearly supported
- system design choices, explainability, human-in-the-loop flows, evaluation logic
- technical breadth across finance, pricing, recommendation systems, matching, and analytics

### What To Avoid

- generic founder language unless the role values it
- long commercial or advisory phrasing that crowds out engineering depth
- unsupported hard claims about years or scale

## One-Page Rules

Keep the final output to one page.

- use only the strongest 2-4 stories
- keep the summary short and directional
- compress earlier experience into one block when needed
- favor dense bullets over long paragraphs
- include metrics only when supported

## Truthfulness Rules

- prefer `senior`, `experienced`, or `nearly a decade` over unsupported year inflation
- prefer `built` only when the project evidence shows hands-on implementation
- prefer `worked with` or `delivered with` when partnership is the truthful framing
- if the JD asks for a technology, verify it in the project before adding it
